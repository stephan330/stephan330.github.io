Git en équipe : Workflow et bonnes pratiques

Git est l'outil de gestion de versions incontournable pour tout développeur. Mais travailler en équipe avec Git demande de suivre certaines conventions et workflows pour éviter les conflits et garder un historique propre.

🌿 Le Git Flow simplifié

Le Git Flow est un modèle de branching qui structure le développement. Voici une version simplifiée adaptée aux petites et moyennes équipes :

  • main — Le code en production, toujours stable
  • develop — La branche d'intégration des nouvelles fonctionnalités
  • feature/* — Branches pour chaque fonctionnalité
  • hotfix/* — Branches pour les corrections urgentes en production

🔀 Workflow pas à pas

1. Créer une branche de fonctionnalité

Toujours partir de develop pour créer une nouvelle branche :

# Se placer sur develop
git checkout develop
git pull origin develop

# Créer la branche feature
git checkout -b feature/ajout-panier

# Travailler et commiter
git add .
git commit -m "feat: ajout du système de panier"

2. Pousser et créer une Merge Request

# Pousser la branche
git push origin feature/ajout-panier

# Sur GitLab/GitHub → Créer une Merge Request
# vers la branche develop

3. Code Review et Merge

Un collègue (ou vous-même sur un projet personnel) revoit le code, suggère des modifications si nécessaires, puis approuve et fusionne dans develop.

4. Mise en production

# Quand develop est stable, merger dans main
git checkout main
git merge develop
git tag v1.2.0
git push origin main --tags

📝 Conventions de commits

Des messages de commit clairs et structurés facilitent la compréhension de l'historique. Voici le format Conventional Commits : 

# Structure
type(scope): description courte

# Exemples
feat(auth): ajout de l'authentification Google
fix(panier): correction du calcul du total
docs(readme): mise à jour des instructions d'installation
style(css): correction de l'alignement du header
refactor(user): simplification du modèle utilisateur
test(api): ajout des tests unitaires pour les produits
chore(deps): mise à jour de Laravel vers v11

💡 Règle d'or : Un commit = une modification logique. Évitez les commits « fourre-tout » comme "correction de bugs" ou "modifications diverses". Soyez précis !

⚠️ Les erreurs courantes à éviter

  1. Ne jamais push sur main directement — Toujours passer par des branches et des merge requests
  2. Pull avant de push — Toujours récupérer les dernières modifications
  3. Ne pas ignorer les conflits — Résolvez-les soigneusement en comprenant les deux versions
  4. Commiter les fichiers sensibles — Ajoutez .env, node_modules/ et vendor/ dans le .gitignore
  5. Faire des commits gigantesques — Préférez des petits commits fréquents

🛠️ Commandes utiles au quotidien

# Voir l'état des fichiers
git status

# Voir l'historique avec graphique
git log --oneline --graph --all

# Annuler les modifications non commitées
git checkout -- fichier.php

# Mettre de côté du travail en cours
git stash
git stash pop

# Rebaser sa branche sur develop
git rebase develop

# Voir les différences
git diff

# Cherry-pick un commit spécifique
git cherry-pick abc123

🔧 Configuration recommandée

# Configurer votre identité
git config --global user.name "Stephan Doreal"
git config --global user.email "ponmetetstephan@gmail.com"

# Activer les couleurs
git config --global color.ui auto

# Définir VS Code comme éditeur par défaut
git config --global core.editor "code --wait"

# Alias utiles
git config --global alias.st status
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.lg "log --oneline --graph --all"

📚 En résumé

Un bon workflow Git en équipe se résume à : branches claires, commits descriptifs, merge requests avec revue de code, et communication avec l'équipe. Ces pratiques vous feront gagner un temps précieux et éviteront bien des maux de tête ! 💪
